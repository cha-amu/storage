스태틱 사이트에 방문자가 글을 남기는 기능을 추가하려면 입력받은 내용을 저장할 곳과 그 데이터를 주고받을 API가 필요하지만, 방명록 하나 때문에 별도 서버를 운영하고 싶지는 않았다. 원하는 조건은 분명했다.

- 지금 사이트처럼 무료로 운영할 수 있을 것
- 방문자가 GitHub 계정 없이 글을 남길 수 있을 것
- 입력창과 목록을 내 사이트에 맞게 직접 디자인할 수 있을 것
- 글을 숨기거나 삭제하는 관리가 어렵지 않을 것
- 작은 기능을 위해 큰 백엔드를 새로 만들지 않을 것

## 외부 댓글 서비스가 맞지 않았던 이유

가장 먼저 떠오른 건 기존 댓글이나 방명록 서비스를 삽입하는 방법이었다. 구현만 생각하면 가장 빠르다. 하지만 방문자에게 특정 계정을 요구하는 서비스가 많았고, 제공되는 화면도 내 사이트 분위기와 잘 맞지 않았다.

`iframe`으로 외부 페이지를 넣는 방법도 비슷했다. 화면의 크기나 여백을 원하는 대로 다루기 어렵고, 모바일 레이아웃까지 자연스럽게 맞추려면 오히려 손이 더 갔다.

Firebase와 Supabase도 검토했다. 둘 다 제대로 된 백엔드가 필요할 때는 좋은 선택지지만, 방명록 하나를 위해 새 프로젝트와 관리 지점을 늘리는 건 과하게 느껴졌다. Supabase는 이미 다른 프로젝트에서 사용하고 있어서, 이 사이트만의 작은 기능은 따로 떼어 두고 싶기도 했다.

그때 떠올린 것이 Google Sheets였다.

## 방명록은 결국 한 장의 표다

방명록에 꼭 필요한 데이터는 많지 않다.

| 작성자 | 내용 | 작성 시간 | 상태 |
| --- | --- | --- | --- |
| 이름 | 방명록 본문 | 작성 시각 | 공개 또는 숨김 |

이 정도라면 데이터베이스부터 꺼내기보다 스프레드시트의 한 행으로 저장하는 편이 단순하다. Google Sheets는 저장소인 동시에 가장 기본적인 관리 화면도 된다. 스팸 글을 지우거나, 내용을 확인하거나, 특정 글을 숨기는 일을 시트에서 바로 처리할 수 있다.

다만 브라우저에서 Google Sheets에 직접 쓰게 할 수는 없다. 시트 접근 권한이나 인증 정보를 프런트엔드에 노출해서는 안 되고, 들어온 값이 올바른지도 서버 쪽에서 확인해야 한다. 이 사이를 연결하는 얇은 API로 Google Apps Script를 사용했다.

Apps Script가 맡는 일은 단순하다.

- 공개할 방명록 목록을 읽어 JSON으로 돌려준다.
- 새 글의 작성자와 내용을 검사한 뒤 시트에 한 행을 추가한다.
- 삭제 비밀번호를 검증하고 글의 공개 상태를 바꾼다.
- 글이 바뀌면 공개 목록 캐시를 갱신한다.

정적 사이트에서는 `fetch()`로 이 API를 호출한다. 외부 화면을 끼워 넣는 것이 아니라 받은 JSON을 직접 렌더링하므로, 입력창과 목록의 모양은 사이트 스타일에 맞게 만들 수 있다.

## 처음 생각한 구조와 실제 공개 구조

기능만 놓고 보면 구조는 아주 간단하다.

```text
정적 사이트
    ↓
Google Apps Script
    ↓
Google Sheets
```

개인 테스트나 읽기 전용 요청이라면 이 정도로도 충분하다. 하지만 누구나 글을 쓸 수 있는 주소를 공개하면 스팸과 반복 요청을 피하기 어렵다. 그래서 실제 공개 사이트에서는 Apps Script 앞에 Cloudflare Worker를 한 겹 더 두었다.

```text
GitHub Pages
    ↓ fetch()
Cloudflare Worker
    ├─ Turnstile 확인
    ├─ 요청 형식 검사
    └─ 반복 요청 제한
    ↓ 인증된 요청
Google Apps Script
    ├─ 입력값 재검사
    ├─ 방명록 행 저장
    └─ 공개 목록 생성
    ↓
Google Sheets
```

브라우저는 Apps Script 주소를 직접 호출하지 않고 Worker를 거친다. Worker는 글쓰기 요청의 Turnstile 토큰과 요청 빈도를 확인하고, 정상적인 요청에만 서버 간 비밀값을 붙여 Apps Script로 전달한다. Apps Script는 그 값을 다시 확인한 뒤 데이터를 저장한다.

핵심 저장 구조는 여전히 Sheets와 Apps Script다. Worker는 공개된 쓰기 API 앞을 지키는 문지기에 가깝다. 작은 개인 사이트의 현재 사용량에서는 이 구성도 각 서비스의 무료 사용 범위 안에서 운영할 수 있었다.

## 글을 읽고 쓰는 흐름

방명록 목록을 열면 정적 사이트가 API에 공개 글을 요청한다. Apps Script는 시트에서 `visible` 상태인 행을 읽어 JSON으로 돌려주고, 프런트엔드는 그 데이터를 사이트 안에 직접 그린다.

새 글을 남길 때는 흐름이 조금 더 길다.

1. 방문자가 이름, 내용, 삭제 비밀번호를 입력한다.
2. Worker가 Turnstile 결과와 요청 빈도를 확인한다.
3. Apps Script가 입력값을 다시 검사하고 시트에 저장한다.
4. 저장된 글을 응답으로 받아 목록에 반영한다.

방문자 입장에서는 평범한 방명록이지만, 운영하는 쪽에서는 별도 서버 프로세스를 계속 띄워 둘 필요가 없다. 관리가 필요할 때는 Google Sheets에서 원본 데이터를 바로 확인할 수도 있다.

## 이 구성이 잘 맞았던 이유

가장 마음에 든 점은 필요한 만큼만 만들 수 있다는 것이다. Google Sheets가 저장소와 기본 관리 도구를 함께 맡고, Apps Script는 그 앞에서 작은 API로만 동작한다. 화면은 정적 사이트가 직접 그리기 때문에 외부 서비스의 디자인에 끌려가지 않는다.

방문자에게 별도 계정을 요구하지 않는 것도 중요했다. 사이트에 들어와 이름과 내용만 적으면 되고, GitHub 계정이 없어도 된다. 방명록 때문에 Firebase나 Supabase 프로젝트를 추가로 관리할 필요도 없었다.

## 분명한 한계도 있다

Google Sheets는 데이터베이스를 완전히 대신하는 도구가 아니고, Apps Script에도 실행 시간과 요청량 제한이 있다. 대규모 서비스이거나 데이터 관계가 복잡하다면 처음부터 전용 데이터베이스와 백엔드를 선택하는 편이 낫다.

무료라고 해서 보안을 생략할 수도 없다. 공개 쓰기 기능에는 입력값 검증, 스팸 방지, 반복 요청 제한이 필요하다. 처음 만든 단순한 구조에 Worker와 Turnstile을 더한 이유도 여기에 있다.

반대로 방문자가 많지 않은 개인 사이트라면 이 한계가 크게 문제 되지 않는다. 나중에 규모가 커지더라도 프런트엔드와 API 사이의 형식만 유지하면 저장소를 다른 백엔드로 옮길 수 있다.

## 마무리

방명록 하나를 위해 처음부터 큰 서버를 만들 필요는 없었다. Google Sheets는 데이터를 보관하고, Google Apps Script는 그 데이터를 읽고 쓰는 API가 된다. 공개 쓰기 요청 앞에는 Cloudflare Worker를 두어 최소한의 안전장치를 마련했다.

결과적으로 서버 관리 부담은 작게 유지하면서, 방문자는 계정 없이 글을 남기고, 화면은 사이트 분위기에 맞게 직접 만들 수 있었다. 이 구성의 장점은 단순히 무료라는 데 있지 않았다. 작은 기능을 작은 구조로 시작하고, 필요한 부분만 나중에 보강할 수 있다는 점이 더 컸다.

근데 어차피 방문자가 없어서 글을 안 남길 것 같다.
